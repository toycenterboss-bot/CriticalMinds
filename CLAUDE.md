# CLAUDE.md — CriticalMinds

Я — Claude, и моя память между сессиями не сохраняется. Memory Bank — единственная связь с прошлой работой над проектом. Его точность = моя эффективность.

## Обязательный ритуал начала КАЖДОЙ сессии
Прочитать все файлы `memory-bank/` в порядке:
1. `projectbrief.md` — что строим и зачем (фундамент, меняется редко)
2. `productContext.md` — принципы продукта (проверять каждую фичу по ним)
3. `systemPatterns.md` — архитектура, паттерны, лог решений
4. `techContext.md` — стек, ограничения, структура репо
5. `activeContext.md` — текущий фокус, ближайшие шаги, открытые вопросы
6. `progress.md` — что готово / в работе / не начато

Без этого не приступать к задачам.

## Когда обновлять Memory Bank
- После значимых изменений: `activeContext.md` и `progress.md` — всегда в той же сессии
- Новое архитектурное решение → запись в лог `systemPatterns.md` с датой и причиной
- По команде «update memory bank» — перечитать и актуализировать ВСЕ файлы
- `projectbrief.md` меняется только по явному решению Андрея

## Иерархия
`projectbrief` → задаёт рамку всем остальным. При противоречии между файлами: brief > productContext > systemPatterns > activeContext. Противоречие — сигнал обновить банк, а не игнорировать.

## Незыблемые правила проекта (дублируют банк, потому что критичны)
1. Куратор никогда не видит содержание дневников и формулировки прогнозов — только метаданные и счётчики. Это правило уровня модели данных/API, не UI.
2. Формат урока неизменен: Крючок (ошибка пользователя) → Концепт (один!) → Проверка → Перенос.
3. Новые уроки = данные для движка, не новый код (кроме принципиально новых типов шагов).
4. Дизайн-токены — только из палитры `C` прототипа; шрифты IBM Plex.
5. Язык продукта и документации — русский.

## Контекст проекта
Приложение для развития критического мышления у взрослых. Первая группа — 5–7 руководителей ИТ-кластера МТС, куратор Андрей. Программа: 12 недель, 36 уроков, еженедельные встречи без преподавателя. Детали — в memory-bank.
